## Project Setup data layer (2026-07-11, Stage 2 — `lib/project-setup/`)

Server-side only. No routes, no UI, no server actions (`"use server"` files) yet — this stage is the module that a future Control Center action layer and a future public `/project-setup/[token]` route will both call.

**Files:** `types.ts` (internal `ProjectSetupRecord` vs. the trimmed `PublicProjectSetupView` returned to token-authenticated callers), `tokens.ts` (raw token generation + hashing), `validation.ts`, `pipeline.ts` (pure status-transition rules), `repository.ts` (the 10 operations below), plus a `.test.ts` file per module.

**Token model.** The raw private-link token is generated server-side with `crypto.randomBytes(32)` (Node's CSPRNG, base64url-encoded — not `Math.random`, not a sequential id, and deliberately not a v4 UUID, which carries less entropy and visually announces its own format). Only `hashRawToken()`'s SHA-256 hex digest is ever written to `project_setups.access_token_hash`; the raw token is never persisted, never logged, and is returned to the caller exactly once, from `createProjectSetup` or `reissueProjectSetupToken`. If that return value is missed, there is no recovery path other than reissuing — this is intentional, not a gap.

**Revocation and reissue.** `revokeProjectSetupAccess` sets `access_revoked_at` only — it never touches `status`, `responses`, or any other lifecycle field, and a revoked Setup still exists and can be read internally via `getProjectSetupByProjectId`. `reissueProjectSetupToken` generates a brand new raw token, overwrites `access_token_hash` (which is the entire invalidation mechanism — there is only ever one hash stored, so the previous raw token stops matching anything the instant it's overwritten), increments `token_version`, sets a fresh `token_created_at`, and clears `access_revoked_at`. It never touches `responses` or any lifecycle timestamp.

**Lifecycle status rules.** `draft → in_progress` (start) `→ submitted` (submit) `→ confirmed` (confirm), with `reopen` returning either `submitted` or `confirmed` back to `in_progress`. Every transition is checked against an explicit allow-list in `pipeline.ts` (`canPerformAction`) — there is no generic "any status to any status" path. `startProjectSetup` is idempotent from `in_progress` (opening the link again mid-way doesn't error) and only ever sets `started_at` the first time. `saveProjectSetupDraft` writes only the `responses` column — there is no parameter through which a caller could smuggle in a status or protected timestamp — and is rejected once a Setup is `submitted`/`confirmed` (reopen first).

**Authorization split.** `getProjectSetupByProjectId`, `createProjectSetup`, `reissueProjectSetupToken`, `revokeProjectSetupAccess`, `confirmProjectSetup`, and `reopenProjectSetup` are internal — they take a `SiteConfig` and re-validate the Project belongs to that property, matching every other Control Center repository's pattern. Role-gating (`getControlCenterRole()`) is intentionally NOT inside `repository.ts` — consistent with every other repository in this codebase, that check belongs in the `actions.ts` layer that will wrap these functions in Stage 3. `getProjectSetupByRawToken`, `startProjectSetup`, `saveProjectSetupDraft`, and `submitProjectSetup` are public — the token is the entire authorization, and their return type (`PublicProjectSetupResult` / `PublicProjectSetupView`) never includes `internalNotes`, `createdBy`, `discoveryId`, `tokenVersion`, `accessRevokedAt`, or anything hash/token-shaped.

**Only one Setup per Project.** Enforced by the migration's unique constraint on `project_id`. `createProjectSetup` checks for an existing row first and returns it as `{status: "exists"}`; if a concurrent create still races past that check, the database's unique-violation (`23505`) is caught and the existing row is returned the same way — mirroring the exact idempotent-retry pattern `lib/inbound/repository.ts` already uses. An `{status: "exists"}` result never carries a raw token, since none was ever stored; reissue instead if a link needs to be sent again.

**What remains for Stage 3 (not part of this stage):** the internal `app/control-center/*/actions.ts` layer (role-gated wrappers around the internal functions above, e.g. a "Send Project Setup" action on a Project), the public `/project-setup/[token]` route and its UI, and the internal Control Center view of submitted responses. No contracts, payments, uploads, or email automation in any stage yet — those remain explicitly out of scope until named otherwise.

**Server actions** (`app/control-center/projects/setup-actions.ts`): `createProjectSetupAction`, `reissueProjectSetupLinkAction`, `revokeProjectSetupAccessAction`, `reopenProjectSetupAction`, `confirmProjectSetupAction`. Every action re-checks `getControlCenterRole()` (owner/editor only — matching every other mutation in this codebase) and resolves the property through `siteRegistry`/`getSiteConfig` before calling the Stage 2 data layer, which independently re-validates the Project belongs to that property. **One-time link behavior.** The Project Setup card never displays a raw link except immediately after a `created` or `reissued` action result — held only in React component state, never written to Supabase, `localStorage`, or the page source, and gone the moment the page is refreshed or the callout is dismissed. Reissuing shows a confirmation dialog explaining that the previous link is invalidated the instant a new one is generated.

Reopen shows a confirmation dialog explicitly warning that reopening does not restore access if it was revoked — reissue separately. Confirm shows a confirmation dialog explicitly stating it does not trigger any contract, payment, or phase change.

**Response review.** `lib/project-setup/response-formatter.ts` turns the free-form `responses` jsonb into readable, grouped fields with no raw JSON: keys are humanized on the fly (`artist_name` → "Artist Name"), booleans become Yes/No, empty/null/undefined values are omitted entirely, arrays of scalars become a comma list, arrays of objects fall back to per-item JSON rather than losing data, and nested objects become their own subsection. ## Vitest environment fix: resolving `import "server-only"` (2026-07-11)

On a real Mac checkout, `npm test` reported 99 passing tests with all 14 failures confined to `lib/project-setup/repository.test.ts`, every one failing before any test body ran with `Cannot find package 'server-only'`.

**Root cause.** `server-only` is not an installed npm dependency of this project at all — there is no `node_modules/server-only` and no entry for it in `package.json`/`package-lock.json`. `import "server-only"` still resolves inside `next build` because Next.js vendors its own compiled copy (`node_modules/next/dist/compiled/server-only`) and aliases the bare `server-only` specifier to it inside its own webpack build. Vitest runs modules directly under Node/Vite with no Next.js webpack build in front of it, so that alias doesn't exist there and the bare specifier can't resolve at all — this has nothing to do with the repository module itself, which is exactly why `npm run build` and `npm run typecheck` were both already clean.

**Fix.** `import "server-only";` in `lib/project-setup/repository.ts` (and every other server-boundary module) is untouched — it stays exactly as it was and keeps guarding the production build the same way it always has. Instead:

- `test/stubs/server-only.ts` — a two-line, no-op stub (`export {};`) used only by the test runner. Contains no production logic and is never referenced by `next.config` or any build output.
- `vitest.config.ts` — added a `resolve.alias` entry mapping the bare `"server-only"` specifier to that stub, scoped entirely to Vitest's own config. Production module resolution (`next build`/`next dev`) never looks at this file.

The same audit checked for `client-only` (Next's other sentinel package) — this codebase doesn't import it anywhere, so no matching stub/alias was added for it, per the "don't add unnecessary aliases" instruction.

**Verification.** `npm run typecheck` — clean. `npm test` — this sandbox cannot execute Vitest at all, before or after this fix, for an unrelated, pre-existing reason: `node_modules/rollup` here only has the macOS-built `@rollup/rollup-linux-arm64-gnu`-shaped binary missing (this sandbox's `node_modules` was installed on a Mac and is platform-mismatched on this Linux arm64 box, and there's no network access here to reinstall). This is the same environment limitation reported at the end of Stage 2 and Stage 3 — it is not something this fix could address, since it's one directory below where the `server-only` resolution even matters. `npm run build` was not re-run in this sandbox for the same reason (it already hangs at the dev-server banner in this environment, unrelated to today's change). All three must be run on your Mac to get a real pass/fail; the fix itself only touches test configuration and cannot affect `next build`'s own module resolution.

## Project Setup: the artist-facing flow (2026-07-11, Stage 4)

Adds the private, token-authenticated `/project-setup/[token]` route — the experience an artist uses after Jonathan has already decided to work with them (distinct from Project Discovery's "should we work together?"). Still no contracts, payments, file uploads, email automation, document generation, or client portal beyond this one setup flow.

**Route and privacy.** `app/project-setup/[token]/page.tsx` is a server component with no Basic Auth (`middleware.ts`'s matcher only ever covered `/control-center`, unchanged here) and no public navigation — `components/site-shell.jsx`'s existing pathname check (which already bypasses the public header/nav/footer for `/control-center` and `/start-your-project`) now also bypasses it for `/project-setup`. `app/project-setup/layout.tsx` sets `robots: { index: false, follow: false, nocache: true }`. This project has no `sitemap.ts`/`robots.ts` at all, so there was nothing to add or exclude there. The raw token is validated server-side on every request by hashing it and looking up the matching non-revoked Setup (`getProjectSetupByRawToken`, unchanged from Stage 2) — there is no other authorization mechanism on this route.

**The single generic unavailable state.** A missing token, a malformed token, a revoked token, and an unexpected server/database error (e.g. a missing migration) are all rendered identically by `UnavailableSetup`: *"This Project Setup link is no longer available. Please contact JMT Music for a new link."* An error is logged server-side only (`console.error`, message text only — the data layer's messages are already hand-written and user-safe, and the raw token is never included), never surfaced to the artist. An invalid link is never distinguishable from a revoked or nonexistent one.

**Public data exposed.** Exactly `PublicProjectSetupView` (unchanged since Stage 2): Setup id, status, `responses`, `completedBy`, `startedAt`, `submittedAt`, and the linked Project's id/title/type plus the Client's artist/contact name. Never exposed: `internalNotes`, `discoveryId`, `propertyId`, `tokenVersion`, `accessRevokedAt`, the token hash, `createdBy`, or any other Project/Client/unrelated data — this was already true of the type itself; Stage 4 adds no new fields to it.

**Config-driven architecture.** `lib/project-setup/config.ts` is the single source of truth for both this route and the Control Center's response review: `SetupQuestion`/`SetupSection` data (not one-off components) define every question's key, label, artist-facing prompt, helper text, type (`single_select`/`multi_select`/`textarea`/`text`), optionality, options, and an optional same-screen follow-up question. `SETUP_LABEL_OVERRIDES` is derived automatically from that same list and passed straight into `response-formatter.ts`'s `labelOverrides` parameter from `project-setup-response-review.tsx` — a label is written exactly once, in `config.ts`, and both the artist review screen and the internal Control Center review render it. An empty, documented `SERVICE_SPECIFIC_SECTIONS` map is the named Stage 5 extension point for Production/Mixing/Mastering/Beat/Session-specific questions — nothing renders from it yet, and the artist experience never shows an empty placeholder screen.

**Screens built (Version 1, same flow for every Project type).** Welcome (personalized via the Client's contact name, falling back to the artist name, falling back to a plain "Welcome." if neither exists) → read-only Project Overview (title, service/type, artist name — no editing of Project identity here) → Communication (single-select + optional "best time" follow-up on the same screen) → Timeline (optional textarea) → Availability (single-select) → Creative References (optional textarea, links as plain text, no uploads) → Deliverables (multi-select, explicitly framed as a planning preference, not a locked-in guarantee) → Final Notes (optional textarea) → Review (grouped exactly into Communication / Timeline and Availability / Creative References / Deliverables / Additional Notes, each section independently editable, unanswered optional fields omitted) → Submit. Visual language deliberately mirrors `app/start-your-project/project-intake.module.css`'s stage/choice-card/review-section structure (including its reduced-motion handling) but scopes its own `--accent`/`--accent-bright` custom properties (emerald, not blue) in `setup-flow.module.css` so it reads as the next stage rather than a re-skin of Project Discovery.

**Draft-save behavior.** Chosen approach: save after each completed step, not debounced autosave — explicitly the simpler option the spec permits, and it keeps request volume proportional to real progress rather than keystrokes. Every "Continue" click calls `saveProjectSetupDraftAction` (a thin wrapper around the unchanged `saveProjectSetupDraft`, which writes only the `responses` column) with the full current answer set before advancing; a failure shows an inline, recoverable message and does not advance the step or clear anything the artist typed; success shows a brief "Saved" note. Going "Back" never triggers a save or clears state — the in-memory answers are the only thing Back touches. The raw token is never written to `localStorage`/`sessionStorage`; it already lives in the URL, and resuming later means the artist opening that same link again, not this app duplicating the token into browser storage.

**Status behavior.** Draft → clicking "Begin Setup" calls `beginProjectSetupAction`, which is what actually flips the Setup to `in_progress` and stamps `started_at` — opening the route alone does not. `in_progress` renders the full interactive flow, pre-filled with whatever's already in Supabase (`hydrateAnswers`, in `setup-flow-helpers.ts`). `submitted` and `confirmed` both short-circuit to a static, read-only screen (`renderModeForStatus` — a submitted or confirmed Setup never reaches the interactive branch of `SetupFlow` at all, so there is no code path that could render it editable). A Control-Center reopen flips status back to `in_progress` server-side, which this component requires no special case for — it's simply interactive again, existing responses intact. A revoked link never resolves to a view in the first place (`getProjectSetupByRawToken` rejects revoked rows before returning), so it always renders through the single unavailable state, not a Setup-aware one.

**Submission.** "Send Project Setup" disables itself, calls `submitProjectSetupAction` (saves the latest answers, then submits — `completedBy` is hardcoded to `"client"` server-side and is never accepted as a parameter from the browser), and only flips local state to `submitted` after a successful result. A failure leaves the artist on the review screen with every answer intact, re-enables the button, and shows: *"I'm sorry—your Project Setup could not be sent just yet. Everything you entered is still here. Please try again in a moment."*

**Security.** The raw token is read once from the route param and passed to server actions as a plain function argument — never logged (`grep`-verified no `console.*` calls anywhere in the new client/action files), never written to browser storage, and the token hash/internal metadata never appear in `PublicProjectSetupView`'s serialized shape (test-verified via `JSON.stringify`). `saveProjectSetupDraftAction`/`submitProjectSetupAction` only ever write the `responses` jsonb column — a payload that tries to smuggle in `status`, `completed_by`, or a protected timestamp lands as inert data inside that jsonb blob and never touches the real columns (test-verified). Response size limits are enforced by the unchanged `validateResponses`/`MAX_RESPONSES_JSON_LENGTH` from Stage 2. `saveProjectSetupDraft`/`submitProjectSetup` both reject once a Setup is `submitted`/`confirmed`. No service-role code (`createSupabaseAdminClient`) is imported by `setup-flow.tsx` or any other `"use client"` file in this route — only by `repository.ts`, reached exclusively through the `"use server"` `actions.ts` boundary.

**Missing-migration behavior.** Identical to the single unavailable state above — `getProjectSetupByRawToken` surfaces its existing migration-hint-shaped error, `page.tsx` logs the message server-side only, and the artist sees the same generic "no longer available" copy. Nothing about a missing table ever reaches the page.

## Content Workspace: schema and data layer (2026-07-11, Stage 1)

Migration `20260711200000_content_items.sql` creates `public.content_items` — the first piece of the approved Content Workspace architecture (`Content Workspace - Architecture & Milestone One Plan (DRAFT).md`). Schema and data layer only: no Control Center routes/pages reference this table yet; the existing `/control-center/content` route is still the old static "Content Studio" stub and is untouched in this stage.

**Apply.** Run `supabase/migrations/20260711200000_content_items.sql` in the SQL Editor after `20260711190000_project_setups.sql`. Then run `supabase/verification/verify_content_items.sql` and confirm: the table and all 29 columns exist; the four RLS policies (staff read, owner all, editor insert, editor update — same posture as every other internal table) are present; the trigger and eight indexes exist; and `project_id`/`client_id`/`beat_id` all report `SET NULL` on delete, not `CASCADE`/`RESTRICT` — a Content Item must outlive whatever it's linked to.

**Why its own table, not the Universal Project model.** A Content Item optionally references a Project, Client, and/or Beat (identical "optional relationship" shape to `projects.client_id`/`projects.beat_id`), but it is not a repurposed `projects` row — the Project model's generic 5-phase lifecycle (`not_started → in_progress → finishing → ready → done`) can't represent an 8-stage editing pipeline without overloading `detail_stage`, and a Content Item isn't an earlier/later stage of a Project the way a Lead is of a Client. There is still no separate `releases` table (Beat and Release remain one row by design, `18 - JMT OS Architecture.md`), so `beat_id` is the release connection for now — a real `release_id` can be added later without touching anything here.

**Status pipeline.** `idea → planned → needs_filming → needs_editing → ready → scheduled → published → archived`, enforced by `lib/content/pipeline.ts`'s `CONTENT_STATUS_TRANSITIONS` map — an explicit, bounded allow-list per status, not a generic any-to-any transition. Reasonable backward correction is allowed at every step before `published` (e.g. `ready` back to `needs_editing`); `published` only ever moves to `archived`, and `archived` only ever moves back to `published` — restoring an archived item is therefore always a single, unambiguous action, never an accidental side effect of some other move. Only `updateContentStatus` may change `status`; the general `updateContentItem` update path never reads a `status` (or `publishedAt`) key from its input at all, the same structural-exclusion guarantee Project Setup's `saveProjectSetupDraft` already uses for `responses`.

**Content type and priority.** `content_type` is a checked, extensible list (19 seeded values plus `other`) — a text column with a check constraint, not a native Postgres enum, so growing the taxonomy is a widened constraint, never an `ALTER TYPE` (matches `clients.stage`'s existing convention exactly). `priority` is `low | normal | high | urgent`, defaulting to `normal`.

**Platforms.** `platforms text[]` constrained via the `<@` array-containment operator against a 9-value allow-list (`instagram_reel`, `instagram_feed`, `instagram_story`, `youtube_short`, `youtube`, `tiktok`, `facebook`, `website`, `email`) enforced identically in `lib/content/validation.ts`, so the database and the app layer can never quietly drift apart. `platform_urls jsonb` holds the actual published destination link per platform once it goes live; both the key and the URL value are validated.

**Assets are metadata-only in Milestone 1.** No Supabase Storage upload pipeline yet — deliberately deferred, per the approved architecture. Each of the six tracked assets (video, audio, artwork, thumbnail, caption, hashtags) has a `*_ready` boolean plus, where relevant, an optional reference field (`*_url`, or `caption`/`hashtags` directly) pointing to wherever the file actually lives — a local drive, a cloud folder, an editor handoff link. Asset reference fields are deliberately **not** validated as strict URLs (a local path is a legitimate value); `platform_urls` values, by contrast, are strictly validated as `http`/`https` URLs, since those are genuinely web destinations. `lib/content/pipeline.ts`'s `computeAssetCompleteness` calculates total tracked / ready / missing / completion percentage against a required-assets list that defaults to all six but accepts an override — there is no per-content-type "smart default" table yet (see Open Questions in the architecture doc: inventing one without real usage data would be presenting a guess as product truth).

**Data layer (`lib/content/`).** `types.ts` (record/input/result types, the four bounded value lists), `validation.ts` (title/content-type/priority/platforms/platform-URLs/notes/scheduled-date/asset-reference/caption/hashtags validation, plus the two functions that normalize a create vs. an update input), `pipeline.ts` (the transition map, active/published/archived selectors, attention-count aggregation, asset completeness), `display.ts` (status/priority/content-type/platform labels — not consumed by any UI yet, provided now so Stage 3 never needs a second copy), `repository.ts` (11 exported functions: `listContentItems`, `getContentItemById`, `createContentItem`, `updateContentItem`, `updateContentStatus`, `archiveContentItem`, `getContentAttentionCounts`, `getScheduledContent`, `getContentItemsByRelatedProject`, `getContentItemsByRelatedBeat`, `getContentItemsByRelatedClient` — every one property-scoped, mirroring `project-repository.ts`'s "one round trip, pure selectors" shape).

**Property scoping and cross-property protection.** Every read/write resolves the property from `site.id` first, exactly like every other repository in this codebase. `createContentItem`/`updateContentItem` additionally verify that any supplied `projectId`/`clientId`/`beatId` actually belongs to the same property before writing — a stray id from another property is rejected outright rather than silently creating a cross-property reference.

**Hard delete.** Not exposed at the app layer. RLS reserves full delete for the owner role directly (`content_items_owner_all`), consistent with every other historical business record in this codebase (`clients`, `project_setups`) never being app-deletable. `archiveContentItem` is the supported "retire this" path, and only from `published` — Milestone 1 defines no other route to `archived`.

**Tests.** `lib/content/validation.test.ts`, `lib/content/pipeline.test.ts`, `lib/content/repository.test.ts` (a hand-rolled fake Supabase client supporting genuine multi-row list queries via `.order()` plus a thenable builder, since — unlike `project_setups`, which has at most one row per Project — `content_items` is a real one-to-many list). ## Content Workspace: server actions (2026-07-11, Stage 2)

`app/control-center/content/actions.ts` — the role-gated `"use server"` boundary between a future Control Center UI and `lib/content/repository.ts`. Still no Dashboard, Pipeline board, detail page, Calendar, asset uploads, AI features, or social integrations — this stage is the action layer only, and `/control-center/content` remains the old static "Content Studio" stub untouched.

**Actions added (5):** `createContentItemAction`, `updateContentItemAction`, `updateContentStatusAction`, `archiveContentItemAction`, `restoreContentItemAction`. No `deleteContentItemAction` — permanent deletion is not an existing pattern in this codebase for historical business records (`clients` and `project_setups` are both owner-only-delete at the RLS layer with no app-level delete function anywhere; `content_items` follows the same posture). `archiveContentItemAction` is the supported "retire this" path.

**Authorization.** Every mutation calls `getControlCenterRole()` and requires `owner` or `editor` — `viewer` and unauthenticated (`null`) are both rejected with the same hand-written message, before the repository is ever touched. The property is resolved the same way every other Control Center action file resolves it: `siteRegistry.find()` confirms the requested property id is real (client-safe summary), then `getSiteConfig()` loads the full server-only config the repository layer requires. An unrecognized property is rejected the same way — before any repository call.

**Validation.** Content-field validation is not duplicated here — `lib/content/validation.ts`'s functions already run end-to-end inside `createContentItem`/`updateContentItem` (Stage 1), and this layer does not re-parse title/content-type/priority/platforms/notes/dates/platform-URLs/asset fields a second time. What this layer owns instead: resolving the property, checking the role, and validating the content item id shape (`isValidUuid`, reused directly from `lib/content/validation.ts`) before ever calling the repository — an id that isn't a well-formed UUID is rejected here, not left for a query to fail on.

**Protected-field handling.** Two layers deep. (1) Type-level: `createContentItemAction`'s input type is `CreateContentItemInput` minus `createdBy` — a well-typed caller cannot even construct a call that supplies one; `updateContentItemAction`'s input type is `UpdateContentItemInput`, which has no `status`/`publishedAt` fields at all. (2) Runtime: a `stripProtectedFields()` helper additionally deletes `status`, `publishedAt`, `propertyId`, `createdAt`, and `updatedAt` from whatever object actually arrives, and `createdBy` is always overwritten with the server-resolved `CONTROL_CENTER_SUPABASE_USER_ID` (or `null`) — never taken from caller input — before either repository function is called. This is deliberately redundant with `lib/content/validation.ts`'s own structural exclusion (Stage 1): the guarantee is now provable at this layer directly, not only one layer down inside a function this layer happens to call.

**Status and `publishedAt` behavior.** `updateContentStatusAction` is the only action that changes `status`, and it does not duplicate the transition rules — it calls `updateContentStatus` (`lib/content/repository.ts`), which itself checks `canTransitionContentStatus` (`lib/content/pipeline.ts`) before writing anything. `publishedAt` is set automatically by that same repository function the first time status reaches `published`, and is never cleared on any later move, including archiving — this action layer does not touch `publishedAt` directly in either direction, only reports back whatever the repository computed. `scheduledAt` is untouched by any status action; it only ever changes through `updateContentItemAction`'s general metadata update, exactly as scoped.

**Archive / restore.** `archiveContentItemAction` moves an item to `archived` — it calls `archiveContentItem` (the Stage 1 repository wrapper, only valid from `published`), not a raw status update, so a caller can't accidentally archive from the wrong state. `restoreContentItemAction` is a distinct, explicitly named action calling `updateContentStatus(site, id, "published")` — the one narrow transition the Stage 1 pipeline allows out of `archived`. Restoring is never exposed as "just another status option" to whatever calls this layer; it has its own name so intent is unambiguous at the call site, matching the "only if intentionally restored" language from the approved architecture.

**Missing-migration behavior.** Identical fixed message across every action — `friendlyMessage()` rewrites any repository error containing "has migration" to: *"Content Workspace is not available until the latest Supabase migration is applied."* No table or column name, no raw Postgrest error, ever reaches the browser. Every action's `catch` block also returns its own hand-written, action-specific fallback message (e.g. "The content item could not be created. Please try again.") if something throws outright rather than resolving — nothing unhandled ever surfaces a raw exception message.

**Revalidation.** Every successful mutation calls `revalidatePath("/control-center/content")` and, where an id is available, `revalidatePath("/control-center/content/{id}")` — safe to call now even though neither route exists yet in its final Stage 3 form; `revalidatePath` on a route with nothing reading this data today is a no-op, not an error.

Covers: owner/editor allowed; viewer and unauthenticated denied on every action; an unrecognized property rejected before any repository call; a malformed content item id rejected before any repository call; valid create (fields passed through, `createdBy` assigned from the server identity or defaulted to `null`, never from caller input); protected-field stripping on both create and update, verified by inspecting the exact object handed to the (mocked) repository call; general update forwarding only the keys actually supplied; a valid and an invalid status transition, both passed through faithfully; `publishedAt` returned exactly as the repository computed it; archive calling `archiveContentItem` specifically (not a raw status update); restore calling `updateContentStatus` with `"published"` specifically; the migration-missing message mapped identically across all five actions; and a raw thrown exception never leaking its message to the caller.

## Bugfix: Growth Engine lead creation was silently broken (2026-07-12)

**Symptom.** Submitting the Add Lead form with completely legitimate data (a real artist name, every optional field left blank) returned only "The lead could not be created." — no diagnostic, and it reproduced on every submission, not just edge cases.

**Root cause — a schema gap, not a validation, payload, or RLS bug.** `public.clients.name` is the table's original pre-Growth-Engine column (`text not null`, no default, from `20260705160000_control_center_schema.sql`). The 2026-07-10 Growth Engine extension (`20260710160000_growth_engine_clients_extend.sql`) introduced `artist_name`/`contact_name` and explicitly deprecated `name` ("Do not write new values here") — but never relaxed its `not null` constraint. `createLead()` correctly followed that deprecation instruction and never set `name` on insert, so every new lead violated a not-null constraint on a column the application was explicitly told not to populate — a guaranteed `23502 not_null_violation`, not a maybe. Corroborating evidence: `20260711180000_inbound_lead_system.sql`'s own Discovery-to-Client conversion function had already been forced to explicitly supply both `name` and `artist_name` on its insert — proof this exact gap was already silently affecting one code path before it was noticed in the Add Lead form. Ruled out during the audit: the Add Lead dialog's field names match `createLead`'s expected FormData keys exactly; `platform`/`project_type` are documented free text with no enforced list (`lib/control-center/lead-display.ts`'s `PLATFORM_OPTIONS` are "UI hints only"), so values like `Piano Lessons (WCMA)` were never being rejected by a controlled list; and the failure is in the insert itself, not authorization — the role check and property lookup both already ran successfully before the insert failed.

**Fix.** `supabase/migrations/20260712090000_clients_legacy_name_nullable_fix.sql` drops `name`'s `not null` constraint. Additive and non-destructive — no existing row is affected (every pre-existing row already has a real `name` value from before the `artist_name` backfill), and it makes the column's actual behavior match its own deprecation comment for the first time. **Must be applied in the Supabase SQL Editor before Add Lead will work** — nothing in application code can substitute for this; the constraint lives in the database, not in `createLead()`.

**Alongside the schema fix**, `app/control-center/growth/leads/actions.ts` was refactored, not redesigned: `createLead` and `updateLead` previously duplicated their own normalization/validation inline; both now call one shared `parseLeadFields()`. Normalization: every scalar optional field (`contact_name`, `email`, `phone`, `project_type`, `budget`, `platform`, `notes`) blank-to-`null`s via the existing `optionalText()` helper, unchanged. `social_links` and `tags` normalize to `{}`/`[]`, deliberately **not** `null` — the actual columns are `jsonb not null default '{}'`/`text[] not null default '{}'`, and writing `null` there would reintroduce a bug of the exact same shape as the one just fixed. Newly added validation, all previously missing: a phone-format check, length ceilings on `project_type`/`platform`/`budget`/`notes`/each tag (soft app-layer guards, not database constraints, since these fields are deliberately free text), and a tag-count ceiling. A new `mapLeadDatabaseError()` replaces the old single blanket message: `23502`/`42703`/`42P01` (not-null/undefined-column/undefined-table — the "migration not applied" family) now return *"Leads are not available until the latest Supabase migration is applied."*; `23505` (unique violation) returns *"A matching lead may already exist for this property."*; anything else still falls back to the original generic message. No raw error code, message, column, or table name ever reaches the browser in any case. `updateLeadStage`, `setLeadFollowUp`, `archiveLead`, and `convertLeadToProject` were not touched — out of scope for this fix.

**Tests.** `app/control-center/growth/leads/actions.test.ts` — a hand-rolled fake Supabase client (thenable query builder, since several call sites in this file `await` an `.update().eq().eq()` chain directly with no terminal `.single()`/`.maybeSingle()`, exactly like the real Supabase JS client's builders). Covers: the exact Daphne Florence scenario (artist name only, every optional field blank, real punctuation/line-break notes, non-listed `platform`/`project_type` free-text values); a simulated `23502` on both create and update mapped to the safe migration-hint message; `42703`/`23505` mapped correctly; an unrecognized error code falling back to the generic message without leaking detail; blank contact name/email/phone/budget/social links/tags all normalizing correctly; comma-separated tag parsing; email/phone/URL validated only when populated; oversized `project_type`/`platform`/`notes` rejected; owner/editor allowed; viewer/unauthenticated denied; an unrecognized property rejected before any database call; cross-property lead access rejected (property scoping); and no duplicate insert on a single successful call.

To verify the real runtime behavior without Vitest, the actual unmodified `createLead`/`updateLead` (only import paths adjusted for a plain Node runner) were run against the same fake-Supabase-client scenarios above via a disposable script; all 11 scenarios passed, including the simulated `23502` regression check, and the script was deleted immediately after. **This sandbox has no live Supabase connection or browser session, so the manual step — actually submitting the Daphne Florence lead through the real Add Lead form after applying the migration — still needs to happen on your Mac.**

This is the "reuse everything, do not create a duplicate system" instruction taken literally where it conflicted with the prompt's own paraphrase of a system it hadn't re-read.

**Route structure**, mirroring the Growth Engine's shape exactly: `app/control-center/content/layout.tsx` + `components/control-center/content-tabs.tsx` (Dashboard / Pipeline / Calendar sub-nav, copied from `growth/layout.tsx` + `growth-tabs.tsx`) → `content/page.tsx` (Dashboard) → `content/pipeline/page.tsx` (Kanban board) → `content/pipeline/[itemId]/page.tsx` (detail — nested under `pipeline`, not a sibling of it, specifically so the Pipeline tab's `pathname.startsWith()` check stays active while viewing a detail page, the same reason `leads/[leadId]` nests under `leads`) → `content/calendar/page.tsx` (Stage 4).

**Content Dashboard** (`content/page.tsx`). One `listContentItems(site)` round-trip; every section below derives from it in memory via `lib/content/pipeline.ts` selectors — no per-section re-query. Seven summary cards (all statuses except Archived, matching the build prompt's own seven-item list). **Attention Required** uses a new pure selector, `selectAttentionRequired` (added to `pipeline.ts`, not a new file): a `scheduled` item whose `scheduledAt` has already passed, or a `ready` item with no `scheduledAt` at all — two narrow, unambiguous rules rather than a scored staleness heuristic, deliberately not reusing `isProjectStale`'s day-threshold shape from `project-pipeline.ts` since Content's pipeline has no equivalent per-status threshold concept yet. **Upcoming Scheduled Content** reuses `selectScheduledWithinDays` (already exported, unconsumed since Stage 1) rather than a second hand-rolled filter. **Recently Published** is a plain sort/slice on `published_at` — simple enough not to warrant its own exported selector. "New Content Item" opens `AddContentItemDialog`.

**Content Pipeline board** (`content/pipeline/page.tsx` + `components/control-center/content-item-card.tsx`). Desktop: horizontally-scrollable 8-column Kanban (`lg:grid-flow-col`). Mobile: stacked, stage-grouped sections, empty stages omitted. Identical structural pattern to `leads/page.tsx`, difference is only that all 8 statuses render as live columns here (including Published and Archived) rather than pulling Archived into a separate `<details>` — because a Content Item's whole life stays in this one table, unlike a Lead which graduates out to a `projects` row. Each card: title, priority badge, platforms, linked Beat/Client (resolved via id→title/name lookup maps built once in the page from `getPropertyBeatLibrary`/`getPropertyClients`, same convention as `growth/page.tsx`'s `clientLabels`), scheduled date. Stage movement is a "Move to" `<select>` calling `updateContentStatusAction` directly (Stage 2, unmodified) — same interaction pattern as `LeadCard`, but improved on it: the select's options are `getAllowedNextStatuses(item.status)` (Stage 1 `pipeline.ts`, already exported, previously unconsumed), not every status unconditionally, so an invalid transition is never offered in the first place rather than round-tripping to the server to be rejected.

**Content Detail page** (`content/pipeline/[itemId]/page.tsx` + `components/control-center/content-detail-actions.tsx`). Read-only metadata/notes/platforms/related-Beat/related-Client/related-Project/scheduled/published display, plus a presence-only asset checklist (six rows, ready/not-ready, no upload control anywhere — Stage 1's own posture, untouched). No inline metadata-edit form was built (not in this stage's requested deliverable list — `AddContentItemDialog` could gain an edit mode later the same way `AddLeadDialog` did, but that's a new feature, not part of finishing what was asked). Pipeline status control: `published` and `archived` each have exactly one allowed next status in the Stage 1 transition map, so those two states get a dedicated, explicitly-labeled button calling `archiveContentItemAction`/`restoreContentItemAction` (Stage 2's purpose-built wrappers) instead of a one-option select; every other status uses the same generic "Move to" select as the Kanban card. `getContentItemById`'s three-way result (`found`/`not_found`/`error`) is handled as three distinct branches — `not_found` is a real 404 (`notFound()`), `error` (e.g. migration not yet applied) renders the same friendly banner every other page uses, which the initial draft of this page collapsed into a single `notFound()` check until caught during polish; a genuine repository error must never present as "this content item doesn't exist."

**Add Content Item dialog** (`components/control-center/add-content-item-dialog.tsx`). Same manual-`pending`/`error`-state, plain-object-argument convention as `ConvertToProjectDialog` (not the FormData+`useActionState` convention `AddLeadDialog`/`AddBeatDialog` use), since `createContentItemAction`'s signature is a typed object, not `(prevState, formData)`. Fields: title, content type, platforms (checkbox group — the field is an array, so checkboxes rather than a single select), related Beat/Client/Project (selects, options passed down from whichever page renders the dialog), notes. No asset fields, no status, no scheduled date — exactly the six fields the build requested, everything else stays at its Stage 1 default (`status: 'idea'`, `priority: 'normal'`). No upload UI. Rendered from both the Dashboard and the Pipeline board (each page fetches its own Beat/Client/Project option lists — no shared client-side cache, consistent with every other Control Center page fetching its own data server-side).

**Content Calendar** (`content/calendar/page.tsx` + `components/control-center/content-calendar.tsx`). New client component — the first calendar UI in this codebase, built from plain `Date` arithmetic (no date library is installed, and adding one for a single grid is out of proportion to the ask). Month view (6×7 grid) and Week view (single 7-day row) toggle; Prev/Next/Today navigation; platform and content-type filters (client-side, since the fetched-once item set is small); clicking a day opens that day's items in a panel below the grid rather than a modal, so it stays simple and keyboard-operable. Items are positioned by `scheduledAt` regardless of pipeline status (a `ready` item can carry a future `scheduledAt` before ever reaching `scheduled`). Planning-only, as scoped: no drag-and-drop, no external calendar sync, no write path from the calendar itself. A sidebar duplicates the Dashboard's "Upcoming Scheduled Content" shape (14-day window via `selectScheduledWithinDays`) for this page's own context.

**One Stage 1/2 cleanup, not a rewrite.** `friendlyMessage()`/`MISSING_SCHEMA_MESSAGE` (Stage 2, `app/control-center/content/actions.ts`) rewrites the data layer's `MIGRATION_HINT` (which names the migration file) into the one user-safe sentence every Control Center surface should show.
